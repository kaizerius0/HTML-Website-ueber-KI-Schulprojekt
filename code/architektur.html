Architektur – So funktioniert diese Website

Diese Seite zeigt, wie unsere Website technisch aufgebaut ist. Wir erklären die Codebase, die verwendeten Technologien, wie wir zu zweit zusammengearbeitet haben und welche Werkzeuge – inklusive KI-Assistenten – wir benutzt haben. Alles, was hier steht, lässt sich im Quellcode auf GitHub nachvollziehen.

Unser komplettes Repository ist öffentlich: github.com/kaizerius0/HTML-Website-ueber-KI-Schulprojekt. Jede HTML-, CSS- und JavaScript-Datei ist dort einsehbar.

// 01 OVERVIEW

Überblick

Die Website besteht aus statischen Dateien – kein Build-System, kein Server-Framework, kein Bundler. Jeder Browser kann die Dateien direkt öffnen, sogar lokal über das file://-Protokoll ohne Webserver. Wir haben uns bewusst gegen Frameworks wie React, Vue oder Svelte entschieden, weil wir HTML, CSS und JavaScript wirklich von Grund auf verstehen wollten.

13 HTML-Seiten
268 Zeilen CSS
~2100 Zeilen JavaScript
3 CDN-Bibliotheken

Was unsere Codebase besonders macht:

  • Komplett statisch – kein Build-Schritt, kein npm install.
  • Kein Framework für die Inhaltsseiten – pures HTML, CSS und JavaScript.
  • Die einzige „große" Abhängigkeit ist Three.js für unser 3D-Spiel.
  • Funktioniert auch ohne Webserver – jede Datei lässt sich direkt im Browser öffnen, was die Entwicklung deutlich vereinfacht.
  • Datenbank-Anbindung (Supabase) läuft per JavaScript-Import aus einem CDN, ebenfalls ohne Build-Pipeline.

<body>

// WICHTIG: Es gibt kein Template-System und keinen Build-Schritt. Header, Nav und Footer sind in jeder einzelnen Datei manuell dupliziert. // 05 MODULES

Die drei größten Module

Zwölf unserer Seiten sind reiner Content – HTML mit Text und Bildern. Drei Seiten sind echte JavaScript-Anwendungen, die wir hier im Detail erklären.

Modul 1: Das 3D-Spiel (spiel.html)

Unter spiel.html liegt ein vollständiges First-Person-Survival-Spiel auf einer prozedural erzeugten Marsoberfläche – komplett im Browser, mit Pointer-Lock, Schießen, Aliens, Quests und einem Flugzeug.

  • Erste-Person-Steuerung mit der Pointer-Lock-API (Maus wird im Spiel eingefangen).
  • Prozedurale Marslandschaft via Simplex-Noise – das Terrain ist eine Plane mit 150×150 Segmenten, deren Höhen aus dem Noise-Algorithmus berechnet werden.
  • 500 Felsen werden über InstancedMesh gerendert (eine Geometrie, 500 Instanzen) – dadurch bleibt das Spiel performant.
  • Aliens mit State Machine: patrol → chase → attack → dead. Wechsel auf Basis von Distanzen und einem „Alert"-Wert.
  • Wellen-System: sinkt die Zahl lebender Aliens unter 5, wird die nächste Welle automatisch gespawnt.
  • Quest-System mit zwei verketteten Aufgaben: Rover reparieren (3× Schrott, 1× Teile) und Alien-Pflanze anbauen (15 s Wachstumszeit, +50 HP bei Ernte).
  • Selbstgebautes Flugzeug aus Three.js-Primitives (Rumpf, Cockpit, Flügel, Propeller) mit eigener Flugphysik.
  • Etwa 1.000 Zeilen JavaScript, eingebettet in der HTML-Datei.

Wir haben Three.js gewählt, weil es die De-facto-Standard-Bibliothek für 3D-Grafik im Browser ist. Geladen wird sie per CDN, ohne lokale Installation:

<script src="https://cdnjs.cloudflare.com/ajax/libs/three.js/r128/three.min.js"></script>
<script src="https://cdnjs.cloudflare.com/ajax/libs/simplex-noise/2.4.0/simplex-noise.min.js"></script>

Modul 2: Das Quiz (quiz.html)

Das Quiz prüft das Wissen zu allen Themen unserer Website mit 15 Multiple-Choice-Fragen. Diese Fragen liegen zentral in quiz-questions.js und werden von beiden Quiz-Seiten geladen (normales Quiz und Live-Auswertung).

  • Vier Zustände als sichtbare/versteckte HTML-Sections: Name eingeben → Start-Screen → Frage-Screen → Result-Screen.
  • Player-Identitäts-System: Jeder Browser bekommt beim ersten Quiz eine eindeutige UUID via crypto.randomUUID(). Diese ID liegt im localStorage und identifiziert den Spieler über mehrere Quiz-Versuche hinweg.
  • Letztes Ergebnis bleibt nach Reload erhalten: Wer die Seite nach dem Quiz aktualisiert, sieht direkt wieder seinen Ergebnis-Screen – kein erneutes Spielen nötig.
  • Antwort-Reihenfolge ist randomisiert via Fisher-Yates-Shuffle, sowohl die Reihenfolge der Fragen als auch die der Antwortoptionen.
  • Insert in Supabase mit allen Daten – inklusive der genauen Antwort-Historie pro Frage, damit die Live-Auswertung die Schwierigkeit jeder Frage berechnen kann.
  • Leaderboard wird client-seitig dedupliziert: pro Spieler-ID nur die Bestleistung.

So sieht der Supabase-Insert im Code aus:

const insertData = {
  name:             state.playerName,
  player_id:        state.playerId,
  score:            state.score,
  total:            QUESTIONS.length,
  percent:          (score / total) * 100,
  duration_seconds: state.duration,
  answers:          state.answeredCorrectly   // [{q: index, correct: bool}, ...]
};
if (state.sessionCode) insertData.session_code = state.sessionCode;

const { data, error } = await supabase
  .from('quiz_results')
  .insert(insertData)
  .select()
  .single();

Modul 3: Die Live-Auswertung (quiz-live.html)

Für den Einsatz im Klassenraum: Die Lehrkraft öffnet diese Seite, generiert einen 4-stelligen Session-Code und projiziert ihn an die Tafel. Während die Schüler das Quiz spielen, sieht die Lehrkraft die Ergebnisse live.

  • Session-Code aus reduziertem Alphabet: ABCDEFGHJKLMNPQRSTUVWXYZ23456789 – ohne I/O/0/1, damit Schüler den Code nicht verwechseln.
  • Polling alle 3 Sekunden: Die Seite fragt Supabase regelmäßig nach neuen Ergebnissen mit diesem Session-Code.
  • Drei States: Setup → Active Session → End-Auswertung.
  • Neue Teilnehmer pulsieren cyan für ein paar Sekunden, damit die Lehrkraft sieht, wer gerade fertig geworden ist.
  • End-Auswertung zeigt Klassen-Statistik (Durchschnitt, Bester, Schlechtester, Schnitt-Dauer) und eine Schwierigkeitsanalyse pro Frage.
// DETAIL: Die Schwierigkeitsanalyse aggregiert bewusst alle Antworten, nicht nur die ersten Versuche. Wenn ein Schüler eine Frage zuerst falsch und beim zweiten Anlauf richtig hat, zählen beide Versuche. So zeigt der Balken sowohl die echte Schwierigkeit der Frage als auch die Lernkurve der Klasse.
// 06 DATABASE

Datenbank: Supabase

Supabase ist eine gehostete PostgreSQL-Datenbank mit automatischer REST-API und einem JavaScript-Client. Wir nutzen es als komplettes Backend, ohne einen eigenen Server zu betreiben – das ist für ein Schulprojekt ideal.

Warum wir Supabase gewählt haben:

  • Wir brauchen eine Datenbank für Quiz-Ergebnisse, die alle Schüler im Klassenraum gleichzeitig schreiben können.
  • Wir wollten keinen eigenen Backend-Server betreiben (Hosting, Wartung, Authentifizierung – alles Aufwand).
  • Supabase macht den direkten Zugriff aus dem Browser möglich, ohne dass wir eine Zwischenschicht programmieren müssen.
  • Das Free Tier reicht für ein Schulprojekt locker aus.
  • Row Level Security (RLS) schützt die Daten auf Datenbank-Ebene – nicht durch das Verstecken von Zugangsdaten.

Warum der Zugangsschlüssel öffentlich sein darf

In unserer Config-Datei steht ein „Anon-Key" – das ist der öffentliche Zugriffsschlüssel der anonymen Rolle. Dieser Schlüssel landet im Browser und kann von jedem im Quellcode gelesen werden. Das ist kein Sicherheitsproblem, weil:

  • Die Row Level Security auf Datenbank-Ebene erlaubt der anonymen Rolle nur das, was sie tun darf: Lesen und Einfügen mit Validierung.
  • Updates und Deletes sind blockiert – niemand kann Einträge ändern oder löschen, selbst mit dem Schlüssel.
  • Selbst wenn jemand den Key kopiert, kann er nichts kaputt machen, was die RLS-Policies nicht ohnehin erlauben.

So sieht unsere lokale Config aus (in .gitignore, wird also nicht committet):

window.QUIZ_CONFIG = {
  SUPABASE_URL:      'https://<projekt>.supabase.co',
  SUPABASE_ANON_KEY: 'eyJ...'
};

Eine Template-Variante (quiz-config.example.js) liegt im Repository als Vorlage zum Kopieren. So kann ein Mit-Entwickler die Datei lokal anlegen, ohne dass wir echte Credentials ins Repo schreiben.

// 07 HOSTING

Hosting: GitHub und Vercel

Wir haben unseren Workflow bewusst so aufgesetzt, wie professionelle Entwickler arbeiten: GitHub für den Quellcode, Vercel für das Hosting.

GitHub – unsere zentrale Plattform für Zusammenarbeit

  • Wir entwickeln zu zweit (Chris und Louis). GitHub ermöglicht uns parallele Arbeit am selben Projekt.
  • Jede Änderung wird als „Commit" mit Beschreibung gespeichert – so können wir jederzeit nachsehen, wer wann was geändert hat.
  • Wenn jemand etwas ändert, lädt er es zu GitHub hoch (git push).
  • Der andere lädt die Änderungen herunter (git pull) und hat sofort den aktuellen Stand.
  • Bei Konflikten – wenn wir beide dieselbe Datei geändert haben – müssen wir die Versionen manuell zusammenführen. Das passierte am Anfang öfter, mit der Zeit immer seltener.
  • Das Repository ist öffentlich – jeder kann den kompletten Code einsehen.

Vercel – automatisches Hosting

  • Vercel ist mit unserem GitHub-Repository verbunden.
  • Sobald wir Code auf den main-Branch pushen, wird die Website automatisch deployed.
  • Vercel macht die Seite weltweit über ein CDN (Content Delivery Network) verfügbar – also schnell aus jedem Land erreichbar.
  • Live-URL: projektkischule.vercel.app
  • SSL/HTTPS wird automatisch verwaltet – wir mussten kein Zertifikat konfigurieren.
  • Keine Server-Konfiguration, keine Datenbank-Migration, keine Wartung.

Unser Deployment-Workflow Schritt für Schritt

  1. Wir ändern Code lokal in unserem Editor.
  2. Commit + Push auf GitHub (git commit -m "..." & git push).
  3. GitHub sendet einen Webhook an Vercel und meldet die neue Version.
  4. Vercel baut die Seite und deployed sie über sein globales CDN.
  5. Live in etwa 30 Sekunden auf projektkischule.vercel.app.

Warum diese Kombination:

  • Die Trennung von Quellcode (GitHub) und Auslieferung (Vercel) ist Industrie-Standard.
  • Beide Dienste haben kostenlose Pläne, die für Schulprojekte mehr als ausreichen.
  • Wir lernen genau den Workflow, den professionelle Webentwickler täglich nutzen.
// 08 AI-ASSISTED DEV

Wie wir mit KI entwickelt haben

Ein zentraler Punkt unseres Projekts: Wir haben den Code nicht alleine geschrieben – wir haben intensiv mit KI-Assistenten zusammengearbeitet. Das ist heute schon gelebte Praxis bei Spotify, GitHub und vielen anderen, wie wir auf der Programmierungs-Seite ausführlich beschreiben.

Was ist Claude Code?

Wir haben vor allem Claude Code verwendet – einen KI-Assistenten von Anthropic, der direkt in unserem Terminal läuft und Dateien lesen, schreiben und ausführen kann. Im Unterschied zu einem normalen Chat-Bot arbeitet Claude Code direkt im Projekt-Verzeichnis und kann mehrere Dateien gleichzeitig anfassen.

Unser Workflow mit der KI

  1. Wir formulieren unsere Anforderungen in natürlicher Sprache (zum Beispiel: „Baue ein 3D-Spiel im Mars-Survival-Stil mit WASD-Steuerung, Schwerkraft und Aliens").
  2. Claude Code generiert den Code, schreibt die nötigen Dateien und erklärt, was es getan hat.
  3. Wir testen das Ergebnis im Browser und melden Fehler oder Wünsche zurück („das Springen fühlt sich zu schwer an", „füge ein Flugzeug hinzu").
  4. In mehreren Iterationen verfeinern wir Features, fügen neue dazu oder räumen Code auf.
  5. Wir lesen, verstehen und passen Teile selbst an – die KI ist Werkzeug, nicht Magie.

Was wir dabei gelernt haben

  • Klare Spezifikationen sind wichtiger als schnelle Anweisungen. Eine vage Anfrage liefert vage Ergebnisse. Je präziser wir formulieren, was wir wollen, desto besser passt der Output.
  • Konzept zuerst, Code danach. Wenn wir erst implementieren lassen und dann nachdenken, entsteht oft Code-Müll, der wieder weggeworfen werden muss. Erst die Architektur durchdenken, dann die KI bitten, sie umzusetzen.
  • Persona-basierte Prompts liefern oft stärkere Ergebnisse. Eine Anfrage wie „Denk wie ein Senior Game Developer und baue mir ein robustes Inventory-System" liefert bessere Architektur als „mach mir ein Inventory".
  • Ohne Verständnis des Codes kann man nicht debuggen. Wenn die KI etwas kaputt macht und wir nicht wissen, was passiert ist, sind wir hilflos. Deshalb lesen wir jeden Teil, den wir generieren lassen.
  • Die KI macht Fehler. Sie erfindet manchmal Funktionen, die nicht existieren („Halluzinationen", wie wir es auf der Grundlagen-Seite beschreiben). Testen ist Pflicht.

Warum bewusst kein Framework

Wir hätten React, Vue oder Next.js nutzen können – die KI hätte das problemlos generiert. Wir haben uns aber bewusst dagegen entschieden:

  • Lernziel: Wir wollen HTML, CSS und JavaScript wirklich von Grund auf verstehen, nicht nur einen Framework-Layer darüber bedienen.
  • Transparenz: Frameworks wie React abstrahieren wichtige Konzepte weg (virtuelles DOM, JSX-Compilation, Bundler). Mit reinem HTML/CSS/JS sehen wir genau, was im Browser ankommt.
  • Einfachheit: Die Seite funktioniert auch ohne Webserver (file://-Protokoll), was Entwicklung und Tests vereinfacht.
  • Wartbarkeit für Schüler: Wer den Code in einem Jahr ansieht, versteht ihn ohne Framework-Vorkenntnisse.
// FAZIT: Die KI hat uns enorm beschleunigt – aber das Projekt wäre ohne unser eigenes Verständnis nicht möglich gewesen. Die Spezifikationen, das Design, die Architektur-Entscheidungen, die Tests – das ist unsere Arbeit. Die KI hat den Code geschrieben, wir haben das Projekt geleitet.
